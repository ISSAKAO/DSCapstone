              Thanks for the feedback. Yes, I do mention dplyr briefly above, and I think it's very promising, in fact just read this today http://blog.rstudio.org/2014/1.... Still not sure how it performs with large datasets, but I know you can connect it to different storage systems.

              Absolutely true that vector is more efficient, and for traditional work with data.frames etc, that is the way to go. However, I was trying to show people that they can automate some of the work they do in R, such as generating a number of graphs automatically, or iterating over the columns in a data.frame. In these cases, the numbers are very short, so loop/vector doesn't make much of a difference.

                              Great post which I have shamelessly linked to from my in-course blog for the students. An alternative to xslx is XLConnect which seems a bit more robust but fails on complex formulae (we have some horrendous student management spreadsheets that I have now put a class together for.)

                              Markup and embedded code is a good way to go - I find Markdown too restrictive so end up with Latex (until Markdown can do colours) for my worksheets. Great for simulations and reports. And they are all, obviously, in version control and hacked using RStudio (default go to analysis package for the life science students here.)

                              One misisng thing - learn how to reshape/rename your data. Very useful.

                                  The one biggy I'd add, and it may be considered "advanced" by some, is to do most of the data munging in your SQL database (if that's your source, of course) and then use PL/R to run your R functions in the database.
