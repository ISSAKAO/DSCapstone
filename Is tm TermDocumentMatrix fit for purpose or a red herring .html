    I noticed that a fair number of people are using the tm package and specificially the associated TermDocumentMatrix data structure.

    I haven't tried using them because I couldn't see how the TDM data structure helps with building predictive models from n-gram counts--it seems to be oriented around associating features (such as bag-of-words frequency counts) to individual documents. Whereas here we are not interested in documents, just phrases, and even then we're not trying to conclude in particular anything about the source documents, we're just using them as fuel for our prediction engine.

    Plus, at first glance, manipulating TDMs seems kind of awkward.

    So, I wonder if there's anything I missed. (e.g., I know there are some utility functions in the tm package) How are you using them, and are you finding them useful or awkward?

    My own approach has been:

    1. Build a big data.frame with one row per line of the 3 source files. Add a source column indicating which data source the line comes from (twitter, blogs, news). Save as an .RDS file. (only takes ~60seconds and at only 1GB the dataframe fits easily into memory, admittedly on a fairly nice machine)
    2. Preemptively shuffle and split up the big dataframe into training, validation, and testing data sets. Archive the validation and testing data sets for undetermined later use.
    3. Make a sample dataframe from the training set consisting of 10k lines from each source.
    4. Using the sample data, build a pipeline to experiment with plausible regexes for various aspects of tokenization. e.g., iteratively refine number-sequence tokenizers by making a dataframe with a context column that has the context surrounding every sequence that tokenizes to <number>, then manually examine the resulting context column for plausibility and missed cases.
    5. With confounding number sequences taken care of, iteratively evaluate sentence-splitting regexes (e.g, insert a `` token in place of the period where you see letters+ period|question mark|exclamation point whitespace+ letters+ whitespace); then do the same thing to insert breaks at the beginning and end of possible quotations.
    6. Split on the <break>s to make a new dataframe with one line per phrase.
    7. Clean up resulting phrases -- remove most punctutation other than . and ', consider lower-casing everything
    8. Compute and tabulate initial rough counts of ngrams on the resulting data. (With refinements like spelling correction, proper name detection, dictionary matching, part of speech determination, stemming, smoothing, backoff considerations, etc. to be handled after data from the rough ngrams are explored and after we have a baseline predictive model)
    9. Iterate.
    10. Apply tokenization and ngram counts to the full training dataset, preparing to wait a long time and work in batches if necessary.

    This of course only gets me as far as n-gram counts, but that's almost the point: once you have a vector of counts, and want to apply smoothing, etc, it seems like you're in standard[*] data-manipulation land, and I'm not sure how tm helps there either? Or am I missing something?

    [*] of course for actual use, you need to index by ngram so they can be looked up quickly, seems like data.table would be helpful here?

    I've managed to use tm and DocumentTermMatrix so far for Task 0, 1, and 2. It's very convenient once you work out a strategy for dealing with the large file sizes.     Patrick GillenSignature Track· a month ago 
    I arbitrarily set the number of folds to 100. Then I set up an outer loop to iterate over each fold. Within that loop, step 1, open the input file en_US.blogs using file(<FILENAME>, "r"); step 2, open an output file using file(paste(output.dir, "en_US.blogs-fold", sprintf("%03d", i), ".txt", sep=""), "w") where i is the iterator variable; step 3, set up an inner loop to write out the lines in the fold. The inner loop is the "clever" part, i.e., it taxed my brain the most! I created a 'folds' list object using the createFolds() function in caret. The inner loop iterated over c(folds[[i]][1]-1, diff(folds[[i]])), again where i is the iterator from the outside loop. Then within the inner loop I issued something like writeLines(scan(file=con.in, what="character", sep="\n", nlines=1, skip=j, quiet=TRUE), con.out). All you have to do then is to remember to close the two dynamic connections at the right places in the two loops.

    So far this strategy has worked. I'm up to Task 5. Some other posters have raised concerns about the size of the TermDocument that results but I've actually not found that a problem on my laptop provided I throw away the sparse terms using removeSparseTerms(). I've used a sparse parameter of 0.5 on the grounds that if a term doesn't appear in at least half the folds then it's probably not too important (especially since the instructors have been encouraging us to use sampling). The other thing you need to do when you get the TermDocument is to collapse it across all the folds because as others have noted it's not really relevant that there are different "documents" in this application. For that I used something like colSums(as.matrix(blogs.3gram)) where blogs.3gram is the TermDocument for 3-grams. For example, after applying all the ready made tools in tm for pre-processing (removing white space, numbers, weird characters, etc.) I end up with more than 7.5 million 3-grams, too many for my laptop to do anything useful with. However, after I filter out the sparse terms I end up with just about 13k and that can be computed with very handily.

    Good luck!

    Ron BjorkSignature Track· a month ago 
    I think one important observation that needs to be made is that the larger the token size, the more combinations of words are possible and so the larger the tdm. Smaller tokens have fewer possible word combinations but larger counts in the source documents and so the tdm is smaller with columns having larger count values in the doc column. So its faster to search bigrams then trigrams from tdm since they are fewer of them. That search of bigrams gives the indexes to docs with minimal size. Note also the a prediction based on a (n+1) gram may not include words that are in a n-gram based prediction, not because those word are not possible but because the dataset you've retained(for deployment) simply was too sparse. Thus use of the back-off approach is needed for getting a estimate of what the (n+1) gram predictor would give for the next word not in the data set of (n+1) grams. Given  these complexities to consider, I think making use of suggested packages to reduce the number of manual tasks, should be carefully considered.

    Quick comment: don't conflate the 3 separate issues:

    • invalid encodings that don't represent any character
    • the existence of non-English text in the "English" corpora
    • how to treat validly-encoded "non-ASCII" (or non-Basic Latin characters that come from other languages, emoji, or ǝʞıן sןooʇ http://www.fliptext.org/.

    Unicode defines 1+ million "code points" that can represent (at least for our purposes) text in "any" language. Separately, there are standards for encoding that Unicode (a sequence of code points) as a particular sequence of bytes. UTF-8 is one such standard. There are others, but UTF-8 it has some nice properties:

    • valid 7-bit ASCII text is valid UTF-8 representing the same text
    • it uses a variable number of bytes per code point, so that it's not necessary to use 4 bytes to represent every character
    • yet it's self-syncing: if you parts of a UTF-8 are corrupted or missing, the text after the corrupted part is unaffected (you don't get frameshift errors that turn the rest of the text into garbage, for example)

    The Wikipedia article on UTF-8 is quite readable.

    But note that not just any sequence of bytes is valid UTF-8! (The high bits in byte 1 of an encoded code point indicate how many following bytes are part of the same codepoint; but in valid UTF-8 those following bytes must start with the bits 10; anything else indicates the start of a new codepoint. See the table.

    Also, systems have to remember what encoding is in effect when they process a string, or else they have to guess. Suppose you had some valid UTF-8 but the system thought it was "Extended ASCII" (standardized as ISO-8859-1). Then, the bytes in the 0x80-0xFF range, which perhaps represent some perfectly grammatical Russian in Cyrillic script (as multi-byte codepoints, remember) would be interpreted as a nonsensical sequence of characters from the second half of this table.

    Our corpora (the en_US/en_US.*.txt files) are encoded (imperfectly) in UTF-8. David Hood helpfully shared a snippet of code that:

    • explicitly sets the Encoding (internal R metadata about each string) to "UTF-8" so that subsequent code doesn't guess encodings incorrectly
    • uses iconv to strip byte sequences which are not valid UTF-8 to '' so that later code doesn't choke on them

    As I recall I got 4 warnings on the terminal when I read in the corpora--I think that means there are 4 invalid UTF-8 sequences in all the original (en_US) files. None of my extensive subsequent processing steps have complained!

    Separately, there is the issue of what to do with characters that are validly encoded Unicode, but are what those of us from the States secretly think are just weird, deformed "proper" letters: Å, é, И, ß, etc. (These should "really" be A, e, N, and B, amirite?) Not to mention this beer I give you: 🍺. (Copy and paste the "square" into RStudio to see what I mean.)

    There's a whole thread elsewhere about strategies for removing these characters. But actually, you can also programmatically normalize (some) of them to proper "typewriter" letters using the stringi package like so: https://gist.github.com/048c55db4e698620a3d7

    Kevin LimburgSignature Track· 12 days ago 
    You should be able to save the TDM as an RDS file. So if your TDM is called myTDM:
    saveRDS("myTDM.RDS",myTDM)

    Then you can load it using:
    myTDM <- readRDS("myTDM.RDS")

    In my current version, I am using a list of data.tables saved as an RDS file to store my clean data and bring it into the app. I have a data.table for each of my ngram counts, so for unigrams there is a table with columns, text and count and so on and so forth up to 5grams. 

    My list of five data.tables only take up about 18MB right now when stored as an RDS file.  I have yet to run into an issue where my application takes too long to find a prediction but my algorithm isnt that complicated right now.
