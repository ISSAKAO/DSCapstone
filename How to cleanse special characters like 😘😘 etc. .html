    Rodrigo Bertollo de AlexandreSignature Track· 7 days ago 
    after you encode to UTF-8 you will find some errors with "tolower" function using the TM package.
    There are other packages that do the same thing and won't give you an error.
    Try searching for packages that will skip different these strings.

    Furthermore, there is a difference between curly and straight apostrophe (like Babatunde said) and also quotes.
    And you will note that when you convert to UFT-8 you won't be able to use gsub to modify this. Therefore, try finding the unicode code for these characters and convert them all to straight so you won't have issues.

    example:

    http://www.fileformat.info/info/unicode/char/2018/index.htm


    there are many unicodes for the same character make sure you get most of them.

    Maher HarbSignature Track· 7 days ago 
    ok, problem solved. I had to pass the encoding argument within DirSource.

    > crps  <- Corpus(DirSource("C:/Coursera-SwiftKey/final/en_US/subset/", encoding="UTF-8"), readerControl = list(language="en_US"))
    I can now search for/replace curly apostrophe's

    > grep("\u2019", crps[[1]]$content[2], value=TRUE)
    [1] "“I don’t know. Maybe they’re getting too much sun. I think I’m going to cut them way back.” I replied."

    > gsub("\u2019", "\'", crps[[1]]$content[2])
    [1] "“I don't know. Maybe they're getting too much sun. I think I'm going to cut them way back.” I replied."

    Hope that helps...

    Maher HarbSignature Track· 7 days ago 

    @Anonymous

    I agree with you that one can discard all non-alphabetic characters and build a model using the large body of text available. Since ordering of words matters, there would be slight loss of information because of removal of sentence-ending punctuation marks. It’s a question of what your strategy for the model is and how far you are willing to go to improve upon a crude model that is solely based on n-gram frequency counts. Given that this’s a learning experience rather than a real project, it might be sensible to start with a not-so-complicated approach that discards all  non-alphabetic characters and bases the modeling on the News dataset only (as the source with most proper English). From my end, I decided to spend couple of days (and no more) on data cleaning. I don’t have high ambitions, but it’s a good learning experience as well and very humbling once you become aware of the complexity in dealing with raw text as data source.

    David HoodCommunity TA· 6 days ago 

    I am reluctant to discard &, feeling it is more appropriate to translate it to and, but I am just kicking off some detailed work on how the various characters are used.

    For the original question of how how to get rid of ALL non UTF-8 characters, something like this should work (keeping in mind that in this box I can't actually add any non UTF-8 characters to the example:

    nzelectorates <- c("Northland", "Ōhariu", "Ōtaki", "Pakuranga", "Palmerston North", "Papakura", "Port Hills", "Rangitata", "Rangitīkei", "Rimutaka", "Rodney", "Rongotai", "Rotorua", "Selwyn", "Tāmaki")
Encoding(nzelectorates) <- "UTF-8" #making sure, though my case my system is UTF-8
justUTF <-iconv(nzelectorates, "UTF-8", "UTF-8",sub='') #swap out any problem character with an empty space

    If you want to investigate the lines you are removing the characters from some more, you could substitute in something other than an empty field, such as "LOOKHEREUTFPROBLEM", grep which entries have the substitute line, and then go back to the original data to see what was there.
